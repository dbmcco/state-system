0 What this is

State System is an organizational state layer: durable working interpretation of what appears true now — and the machinery to keep it current, evidence-backed, freshness-aware, and governance-gated. This diagram is the engineer's reference for the layer stack, the update data-flow, and the boundaries that keep interpretation, integrity, and authorization from collapsing into one another.

🧠 vs 🔒 The model/code split

The model owns interpretation (what changed, what matters, what's uncertain). Code owns integrity (schemas, evidence refs, permissions, persistence, audit, execution safety). No thresholds, no hidden routing — every model-mediated decision is inspectable.

📖 Journal = truth · Snapshot = view

The append-only journal is the source of truth. Snapshots are materialized current views. Never update a snapshot without a journal entry; corrections are new entries, so full replay is always possible.

🚦 Three independent gates

Preflight (reachable?) · Freshness (recent?) · Governance (authorized?). One does not imply the others — a fresh, reachable source still cannot authorize an external action.

🧭 Durable substrate vs working models

The substrate is truth-seeking (objects, claims, evidence, accepted state, journals). Working models are task-seeking projections (packages, packets) consumed by agents, apps, reports. Consumers never mutate the substrate directly.

2 Update data-flow — how a change becomes durable state

The five-phase loop crosses every layer. Two invariants hold throughout: the journal is the source of truth, and the model decides meaning while code enforces contracts. Idempotent around trigger / proposal ids.

trigger → durable state → propagationeach box maps to a layer in the stack above
PHASE 1 · L1–L2
Notice
Receive & dedupe a source event/trigger (idempotency key); build an evidence packet of refs, not blobs.
▶
PHASE 2 · L6
Interpret
Model reviews evidence + snapshots + persona → emits a journal proposal (patch, evidence refs, uncertainty, proposed actions) — or none.
▶
PHASE 3 · L7→L4
Commit
Committer validates schema/authority/evidence/risk → appends immutable journal entry, materializes snapshot.
▶
PHASE 4 · L4→L5
Propagate
Queue rollups for parents (same lifecycle); emit review signal; surface follow-up opportunities.
▶
PHASE 5 · L5→L8
Index
Record accepted commits in the recent-change registry; re-package for agents & consumers.

Commit outcomes L7

✓ Accepted

append journal · persist memory · materialize snapshot · queue rollups · emit signal

⏳ Pending

valid but needs approval — no protected mutation, no risky action; record pending proposal

✗ Rejected

invalid/unauthorized/unsafe — no append, no write; emit rejected + reason

∅ No-op

model proposes nothing — append nothing; emit no_update_warranted for audit

Approval posture L7

Not every update needs human approval:

  • Auto-commit — low-risk internal interpretations when evidence & permissions are sufficient.
  • Requires approval — external commitments, protected mission changes, policy, relationship-sensitive claims, business-risk actions.
  • Approval attaches to the journal entry or proposed action — it never disappears into chat context.

Captured agent responses are durable artifacts, not accepted state — they enter state only via the normal review / governance / commit path.

3 The boundaries that prevent false confidence

Three independent gates
Preflight (reachable?) is checked at L2. Freshness (recent?) is checked at L2. Governance (authorized?) is checked at L7. They are separate by construction: a reachable, fresh source cannot authorize an external action, and a governed action still needs fresh evidence. Conflating them is exactly how agents act on stale, unproven, or unauthorized state.
Product vs. instance
The product repo (this one) holds schemas, runtime code, contracts, fixtures (SampleCo, ResearchCo, PortfolioCo). A deployed instance owns live records, credentials, local paths, source-owned & State-System-owned indexes, generated package exports. Agents consume rendered artifacts — they never touch the runtime root. This is what makes the product safely open-source.
Federation boundary
InstanceFederationPack is the boundary object for governed cross-instance reads — it does not sync raw data. Default local_materialization=false: identity stays remote, outputs are safe summaries with evidence refs, subject-notes may demote/explain but never hidden-filter, and a repair policy says what to do when the route is stale. A company may read personal relationship evidence only through an explicit governed route — never by copying raw personal records.

Model-mediated update rule L6

Code gathers factual inputs and exposes update tools. The model decides what changed, which family relationships matter, what is uncertain, and whether a proposed update is worth recording. The schema makes those decisions explicit after they are made — it never hides them inside prompt prose.

Semantic retrieval that returns explicit State System records (claims, evidence, memory, journals, operating pictures) with provenance is part of the integrity layer — evidence plumbing, not hidden organizational judgment.

Write boundary L7–L8

Agents may propose updates, record evidence, or trigger refresh when a runtime grants that capability. They must not:

  • directly mutate source-owned truth or external systems (unless the package exposes an allowed action and governance permits);
  • materialize raw federated data unless a pack explicitly permits it;
  • infer behavior from connector names — connector behavior lives in source modules + tool-action contracts;
  • treat a stale package as permission to act externally.

4 Substrate vs. working models — the "Gordian knot"

One substrate, many projections. This is why State System does not grow a separate architecture for CRM, wiki, company memory, trips, meetings, creative arcs, code review, and agent briefings — those are all working-model profiles over the same substrate.

Durable substrate — truth-seeking

objects · claims · evidence · accepted state · journals · memory · governance

The immutable, scoped, evidence-backed record. Append-only. Replayable.

Working models — task-seeking

bounded projections over the substrate for a purpose

ContextPackage · ModelReviewPacket · AgentActivation — persona/app scoped, freshness-stamped.

Consumers

agent-runtime packets · CLI agents · apps · reports · wiki views · CRM views · model reviewers

Read rendered artifacts. Never mutate the substrate directly.
