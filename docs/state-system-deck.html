Why this exists

Agents can't answer "what is the current state?"

The evidence is scattered across dozens of systems. Some are fresh, some stale, some unreachable, some only declared, some governed elsewhere. Without an explicit state layer, agents either over-trust whatever they saw last — or hide uncertainty in prose.

Scattered

Docs, messages, tasks, CRM, calendar, repos, memory, local runtime artifacts — no single current view.

Mixed freshness

Some sources fresh, some stale, some failed, some only planned. Recency is usually invisible.

Unauditable

Interpretation, fact, and permission blur together inside chat context. No provenance. No replay.

Ungoverned

Protected actions fire off hidden thresholds and "best guesses" instead of an explicit authorization boundary.

State System makes the boundary between what appears true, why it changed, what's fresh, and what's allowed explicit and inspectable.

Core idea

State is not a note, a prompt, or a context dump.

It is a durable, scoped record of what appears true now — and the machinery to keep it honest.

State is a durable record of…

  • What appears to be true now
  • Why that view changed
  • Which evidence supports it
  • Which sources are reachable & fresh
  • What is stale, failed, unknown, planned, or federated
  • What needs attention
  • Which actions are proposed, approved, blocked, or taken

Working interpretation, not absolute truth

State System does not claim to hold ground truth. It holds current working interpretation: what appears true now, why we believe it, what evidence supports it, what remains uncertain, and what action may be warranted.

Every record is scoped (instance / domain / persona), evidence-linked, and freshness-stamped — so a reader never confuses a stale view with permission to act.

How a change becomes durable state

The update lifecycle — a five-phase loop

Two invariants: the journal is the source of truth, and the model decides meaning while code enforces contracts. The twelve detailed stages collapse into this loop.

PHASE 1 · SOURCE
Notice
Receive & dedupe a source event / trigger; gather an evidence packet (refs over blobs).
▶
PHASE 2 · MODEL
Interpret
Model reviews evidence + snapshots + persona → emits a journal proposal (or none).
▶
PHASE 3 · GATE
Commit
Committer validates schema, authority, evidence, risk → appends immutable journal entry, materializes snapshot.
▶
PHASE 4 · CASCADE
Propagate
Queue rollups for parents; emit review signal; surface follow-up opportunities.
▶
PHASE 5 · INDEX
Index
Record accepted commits in the recent-change registry for human & agent review.

Update classes

Direct · Interpretive · Corrective · Rollup · Developmental

Journal = truth

Immutable. Corrections are new entries, never edits. Full replay possible.

Snapshots

Materialized from history. Compact reads; drill into journal for provenance.

Idempotent

Replayed triggers never duplicate journal entries or memory writes.

Approval posture

Low-risk interpretations auto-commit; external / protected changes stay pending.

Evidence & provenance

Every claim is traceable — to a reference, not a blob

State System stores pointers to source material, not duplicated content. Large material stays in its system of record; state keeps the link and the interpretation.

pointers Evidence references

  • meeting note id
  • email / chat thread id
  • task / document id
  • metric snapshot id
  • prior journal entry id

judgment Interpretation vs. fact

The model may record interpretations — "the campaign narrative feels diffuse" — and persist them. But it must label uncertainty.

External facts — "the client approved the proposal" — require evidence before they're stored as fact.

rule The boundary rule

Low-confidence interpretations are allowed to persist, but the system never converts interpretation into fact without evidence.

Captured agent output is durable evidence — it becomes state only through the normal review, governance, and commit path.

Provenance is a first-class column: every snapshot points back through the journal to the exact trigger, actor, and source material that produced it.

Freshness — prove recency, don't fake it

A status enum with hard non-conflation rules

StatusWhat it meansAgent behavior
freshRecord carries source- or ingestion-owned recency within policy.Safe to use for operational answers.
stalePast stale_after / expired validity window.Refresh before external action. Usable with caveat.
failedAdapter ran but couldn't prove recency (error / unreachable).Visible gap. Don't pretend it's current.
unknownNever checked; no recency evidence.Treat as unproven. Caveat explicitly.
plannedDeclared in capability, adapter not implemented yet.Remains a declared gap, never silently usable.

probe_only ≠ fresh

A probe that ran proves availability only — not that the corpus or index is current.

package_generation ≠ fresh

Package generated_at is export time, not source recency. Never promote it to fresh.

declared_gap ≠ fresh

Unavailable / auth-blocked / disabled / not-configured sources stay visible with a reason.

Aggregates may be usable while still exposing stale/unknown children — child gaps must remain visible in read models and packages.

The separation that prevents false confidence

Three independent questions — one does not imply the others

Reachability, recency, and authorization are checked separately. Conflating them is how agents act on stale, unproven, or unauthorized state.

preflight

Is it reachable?

Proves live access to a connector — or records why access isn't proved. The adapter actually connected.

freshness

Is it recent?

Proves source recency within policy — or records stale / failed / unknown / planned. The watermark is current.

governance

Is it authorized?

Authorizes protected external action — independent of access and recency. A separate boundary.

A reachable, fresh source still can't authorize an external action. A governed action still needs fresh evidence. Each gate stands alone.

Committer & governance

The committer turns proposals into durable effects

It never reinterprets business meaning. The model proposes; the committer decides what is valid, allowed, persisted, pending, or rejected — with no silent fallback.

accepted

  • Valid + no approval needed
  • Append journal + persist memory
  • Materialize snapshot, queue rollups
  • Emit review signal

pending

  • Valid — needs human approval
  • No mutation of protected state
  • No risky action executed
  • Emit pending_approval

rejected

  • Invalid / unauthorized / unsafe
  • No journal append, no memory write
  • Emit rejected + reason

no-op

  • Model proposes nothing
  • Append nothing
  • Emit no_update_warranted for audit
The gate checks: schema validity · actor authority · write permission · evidence existence · action risk · protected-state rules · duplicate ids · memory/promotion policy. Each check produces an explicit result.

Federation — governed cross-instance reads

Query across instances — without copying raw data

Personal instance

Owns personal interpreted state & relationship substrate.

  • health, activity, media
  • personal relationships
  • owned indexes

🛡️ InstanceFederationPack

The boundary object for governed cross-instance queries. It does not sync raw data.

  • identity boundary — subjects stay owned by the remote source
  • materialization — local_materialization = false by default
  • freshness policy — checked-at, watermark, stale-after, gap refs
  • subject-note policy — demote / explain, never hidden filter
  • output policy — safe summaries + evidence refs, no raw corpora
  • repair policy — what to say when the route is stale / missing

Company instance

e.g. SampleCo — owns company interpreted state.

  • deals, campaigns, projects
  • operating pictures
  • governance & approval state
A company may read personal relationship evidence only through an explicit governed federated route — it must never copy raw personal records or treat personal sources as company sources.

Agent integration contract

Agents read rendered packages — they don't scrape roots

✓ Agents should

  • Read InstanceAgentPackage / the exported read model
  • Inspect freshness before answering (status, gaps, expired refs, stale-after)
  • Use explicit question routes & tool/action refs from the package
  • Reference source module contracts for connector behavior
  • Treat captured agent output as evidence for review
  • Keep governance separate from freshness & preflight

✗ Agents must not

  • Infer behavior from connector names (drive, msgvault, linear)
  • Scrape private runtime roots or raw corpora
  • Materialize raw federated data unless a pack explicitly permits it
  • Directly mutate source-owned truth or external systems
  • Treat a stale package as permission to act externally
  • Authorize protected actions from freshness or preflight alone
For agent runtimes: ship a rendered scope packet in the prompt, expose refresh / understanding / search as tools, and point AGENTS.md at the package — never duplicate its contents.

Pressure tests & conformance

Packages must be answerable, not just present

Pressure questions are real operational questions with structural assertions against package JSON — they don't grade answer text, they check that a package exposes enough structure to answer responsibly.

Package pressure

  • Does the route exist for this question?
  • Is required source coverage present?
  • Are the right tool/action refs exposed?
  • Are freshness gaps & answer policy visible?
  • Are federation boundaries declared?

Ecosystem conformance

  • Capability connectors have source modules
  • Tool actions ref known modules + types
  • Route tools have action contracts
  • Route source-module refs exist
  • Packages emit linkage fields

North-star answer

A deterministic read surface turns the question set into a grounded substrate — current state, why, recent changes, evidence, uncertainty, responsibility, next actions — without copying raw corpora or authorizing execution.
