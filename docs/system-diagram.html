State System Orientation Diagram

State System is the organizational state layer: it turns scattered signals from known and unknown sources into durable, evidence-backed interpretation that humans and agents can act from. This diagram shows the system as of May 1, 2026, including source generalization, coherence maintenance, what is implemented, what is partial, and what should come next.

System Map

Workflow Explainer

1. Source Generalization Loop

  1. A known or unknown source is described by a source recipe: kind, identity, dedupe, permissions, sync behavior, and confidence.
  2. Canonical sources such as Gmail and Google Drive are distinguished from mirrors and indexes such as msgvault, Folio tenant corpora, and pgvector.
  3. Indexes find candidate evidence, but the system only trusts items that resolve back to stable source refs.
  4. Each observation is classified as archive, evidence only, memory candidate, state review trigger, task proposal, approval item, or activation context.

2. State Update Loop

  1. A source system emits a signal: a deal moves, a commit lands, a task changes, or a human makes a decision.
  2. The signal becomes a source event with stable refs, idempotency, evidence, and watermarks.
  3. The runner builds a review packet with the relevant state, memory, persona, evidence, and governance constraints.
  4. A model or fixture interprets what changed and proposes state, memory, or action effects.
  5. The committer validates the proposal, enforces governance, and persists accepted journals, snapshots, memory, review signals, and rollup requests.

3. State Coherence Loop

  1. After a governed commit, the system checks what existing state, memory, rollups, packages, and activation outputs may now be stale.
  2. Identity resolution prevents duplicate people, companies, repos, documents, agents, and concepts from fragmenting state.
  3. Compact metadata lets agents scan state cheaply before opening detailed records.
  4. Review cadence catches stale, uncertain, orphaned, sensitive, or approval-blocked state before it pollutes activation.

4. State Activation Loop

  1. A human or agent needs to act from state: “What is going on?” or “What should Laura know?”
  2. The system scopes the need to relevant state objects, journals, memory, evidence, recent changes, and governance constraints.
  3. For agents, the current implemented path is a persona context package plus narrative rendering.
  4. For humans, the next contract should produce a grounded answer with refs, uncertainty, freshness, and allowed next actions.
  5. Both surfaces should use the same underlying state, not two separate knowledge systems.

5. Feedback Loop

  1. A human or agent response is captured as an artifact, not accepted as truth by default.
  2. The captured response can become a new source event or review input.
  3. The model decides whether it proposes a state update, memory write, task, approval request, or no-op.
  4. Governance determines whether the proposal can become durable state or action.
  5. This closes the loop without letting downstream reasoning bypass evidence or approval boundaries.

Where We Are

Strong spine already exists

  • File-backed stores, schemas, examples, and validation harness.
  • Source events, triggers, idempotency, and Git commit source adapter.
  • msgvault and Folio/GDrive now clarify the need for source recipes, mirror/index refs, and tenant-scoped evidence access.
  • Review packets, fixture model outputs, governed committer, journals, snapshots, memory entries, and review signals.
  • Recent-change registry, persona context packages, CLI access, and agent narrative rendering.

Next clarity move

  • Define Source Generalization and Evidence Access so future sources inherit identity, dedupe, permissions, sync, and evidence rules.
  • Define State Coherence Maintenance so new durable state can reweave affected rollups, packages, identities, and stale records.
  • Then define the State Activation Contract that unifies human answers and agent packages.
  • Defer API, database, MCP, live models, and sync workers until the activation contract is boring and useful.
The center of gravity should remain durable organizational interpretation. Source indexes find possible evidence, coherence keeps state from decaying as sources expand, and context packages, answers, and agent prompts are activation surfaces over that state, not replacements for it.
