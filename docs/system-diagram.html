State System Orientation Diagram

State System is the organizational state layer: it turns source-owned evidence, explicit connector contracts, and instance readiness records into CLI-ready agent packages and governed federation routes. This diagram shows the system as of May 19, 2026, including the generic SourceModuleSpec, ToolActionContract, and QuestionRouteContract layer; instance capability, preflight, freshness, and readiness records; b-state/Samantha and LFW/Caroline packages; Navicyte/Helena and Synthyra/Ingrid scaffolds; and federation packs that do not materialize raw remote data.

Application Substrate Map

The new application repos should build against the same proposal, approval, evidence, and state substrate. Apps own their workflow loops; State System owns durable interpreted state and accepted doctrine. Qualitative judgment remains model- and human-mediated.

Workflow Explainer

1. Source Module Contract Loop

  1. A connector type is described by `SourceModuleSpec`: source ref shape, access owner, module modes, preflight, freshness, index ownership, output policy, and gap behavior.
  2. `ToolActionContract` maps canonical tool actions to deployment-specific backing tools without hiding mode differences such as live API, local sync, export, or historical cache.
  3. `QuestionRouteContract` names required source coverage, tools, answer policy, fallback behavior, and federation requirements before an agent answers.
  4. Packages expose these contract refs directly so behavior is visible in schemas and examples, not private prompts.

2. State Update Loop

  1. A source system emits a signal: a deal moves, a commit lands, a task changes, or a human makes a decision.
  2. The signal becomes a source event with stable refs, idempotency, evidence, and watermarks.
  3. The runner builds a review packet with the relevant state, memory, persona, evidence, and governance constraints.
  4. A model or fixture interprets what changed and proposes state, memory, or action effects.
  5. The committer validates the proposal, enforces governance, and persists accepted journals, snapshots, memory, review signals, and rollup requests.

3. State Coherence Loop

  1. After a governed commit, the system checks what existing state, memory, rollups, packages, and activation outputs may now be stale.
  2. Identity resolution prevents duplicate people, companies, repos, documents, agents, and concepts from fragmenting state.
  3. Compact metadata lets agents scan state cheaply before opening detailed records.
  4. Review cadence catches stale, uncertain, orphaned, sensitive, or approval-blocked state before it pollutes activation.

4. Instance Package Loop

  1. An instance capability pack declares source connectors, indexes, tool bindings, governance, runtime constraints, and agent/persona surfaces.
  2. Preflight and source freshness records prove live access, local sync, historical cache bounds, or explicit planned gaps.
  3. The package renderer emits CLI-ready agent context with source module refs, route refs, tool action refs, freshness status, gap refs, and repair guidance.
  4. b-state/Samantha and LFW/Caroline render as daily-use packages; Navicyte/Helena and Synthyra/Ingrid render as scaffolded packages with readiness gaps.
  5. Activation and report surfaces consume packages as bounded context, not as durable truth.

5. Feedback Loop

  1. A human or agent response is captured as an artifact, not accepted as truth by default.
  2. The captured response can become a new source event or review input.
  3. The model decides whether it proposes a state update, memory write, task, approval request, or no-op.
  4. Governance determines whether the proposal can become durable state or action.
  5. This closes the loop without letting downstream reasoning bypass evidence or approval boundaries.

6. Capability And Freshness Loop

  1. Capability context does not prove live access and does not authorize action; preflight proves reachability and governance authorizes protected effects.
  2. Garmin Connect is exposed as a fresh local-sync source in the personal package.
  3. Spotify is exposed as a usable historical cache with a stale live-API gap, not current listening truth.
  4. LFW Linear and GitHub are fresh from source-owned watermarks; LFW transcript raw and processed pipelines remain planned gaps.
  5. PAIA memory is a generic source module with separate personal connector surfaces for Samantha and owner memory.
  6. No private records, raw corpora, credentials, or absolute runtime paths belong in public diagrams or examples.

7. Federation Loop

  1. `InstanceFederationPack` declares governed cross-instance query boundaries and the route, tool action, source module, freshness, and output policies for them.
  2. Personal b-state can read LFW interpreted state through an instance read route.
  3. LFW can query personal Relationship Substrate through a governed relationship route without materializing raw personal relationship records.
  4. Portfolio rollups across Navicyte and Synthyra remain planned until their packages prove readiness.
  5. Federated answers carry safe summaries, evidence refs, watermarks, gap refs, and repair guidance rather than raw remote corpora.

Where We Are

Strong spine already exists

  • File-backed stores, schemas, examples, and validation harness.
  • Source events, triggers, idempotency, and Git commit source adapter.
  • msgvault and Folio/GDrive now clarify the need for source recipes, mirror/index refs, and tenant-scoped evidence access.
  • Review packets, fixture model outputs, governed committer, journals, snapshots, memory entries, and review signals.
  • Recent-change registry, persona context packages, CLI access, agent activation records, activation rendering, and captured agent responses.
  • Trace manifests and `trace-run` now replay accepted-state, approval-gated, agent-activation, and stale-context refresh flows with inspectable artifacts and a validation report.
  • The operational loop now runs source/evidence -> review -> commit -> package -> activation -> captured response with an operator summary.
  • The approval-gated trace proves pending external action can be held without materializing state or executing the action.
  • The agent-activation trace proves an agent can receive action boundaries and respond without its response becoming truth by default.
  • The stale-context trace proves expired packages surface validity, refresh blockers, prohibited external actions, and captured refusal before external use.
  • The mission-control design is committed and queued as `state-mission-records-v0` behind the app handoff lane.
  • Schema-valid app fixture chains now cover six app-substrate flows: Prospect/Outreach/CRM, Meeting, Thoughtforge, Visual Forge, and CRM outcome doctrine.
  • Company memory and CRM operating picture now have a first JSON substrate read model with Laura and Patrick package slices.
  • Company capability packs now seed runtime State System records for LFW, Synthyra, and Navicyte before PAIA tool wiring.
  • Generic source module, tool action, and question route contracts are schema-backed and tested.
  • Instance agent packages are CLI-ready for b-state/Samantha and LFW/Caroline, with Navicyte/Helena and Synthyra/Ingrid scaffold packages carrying typed readiness gaps.
  • Federation packs declare no-raw-materialization boundaries for personal-to-LFW, LFW-to-Relationship-Substrate, and planned portfolio rollups.
  • Relationship Substrate is represented as a governed source module and federated route, not a b-state-only assumption.
  • PAIA memory is represented as `source_module.paia_memory` with separate b-state connector surfaces for Samantha and owner memory.
  • `paia-bootstrap-export` now refreshes the PAIA company capability, preflight, and source freshness read-model artifact layout.
  • `source-heartbeat-run` makes State System the active freshness orchestrator, with direct local metadata checks and explicit delegated connector boundaries.
  • `state-root-migrate` moves the canonical runtime root into the visible LFW workspace with copy-not-destructive semantics and a compatibility symlink from the old hidden path.

Next clarity move

  • Keep Spotify live OAuth repair separate from the historical-cache stale package contract.
  • Repair LFW transcript raw and processed pipelines before any package claims transcript evidence is current.
  • Promote Navicyte/Helena and Synthyra/Ingrid from scaffold to ready only after source-owned preflight, freshness, and index records exist.
  • Have PAIA consume runtime read models through connector preflight and governance checks instead of app-local company specials.
  • Keep wiki, dashboard, and report surfaces as projections over JSON substrate, not as durable truth sources.
  • Keep package pressure questions as real route-contract checks for source coverage, fallback behavior, federation, and gap visibility.
  • Define State Coherence Maintenance so new durable state can reweave affected rollups, packages, identities, and stale records.
  • Extend reporting surfaces for human inspection without making humans direct state-system operators.
  • Defer API, database, MCP, live models, and sync workers until the activation and reporting contracts are boring and useful.
The center of gravity should remain durable organizational interpretation. Source indexes find possible evidence, coherence keeps state from decaying as sources expand, and context packages, agent activations, captured responses, and reports are surfaces over that state, not replacements for it.
